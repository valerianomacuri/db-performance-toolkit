# 🧠 Database Performance Toolkit

This project demonstrates **how databases actually work**, not just how to write ORM code.

It focuses on detecting, analyzing, and proving query performance improvements.

Technologies used:

* Node.js
* TypeScript
* TypeORM
* PostgreSQL

---

## 🎯 Purpose

This project shows:

* how to detect slow queries
* how to analyze execution plans
* how indexes change query behavior
* how ORM-generated SQL should be inspected
* when and why to use materialized views
* how to compare performance **before and after optimizations**

It is about **engineering judgment**, not features.

### 1️⃣ Exact WHERE vs bad filters

```bash
npm run db:where
```

File: `01-exact-where.ts`

Demonstrates:

* exact `WHERE` using indexed columns
* why `LIKE '%value%'` causes `Seq Scan`
* execution plan differences using:

```sql
EXPLAIN (ANALYZE, BUFFERS)
```

---

### 2️⃣ Controlled joins (ORM best practice)

```bash
npm run db:joins
```

File: `02-controlled-joins.ts`

Shows:

* avoiding `relations`
* selecting only required columns
* preventing unnecessary joins

---

### 3️⃣ Heavy query baseline (no optimizations)

```bash
npm run db:heavy
```

File: `03-heavy-query-baseline.ts`

This is the **baseline**:

* aggregation over large datasets
* no indexes
* no materialized views

Used for comparison.

---

### 4️⃣ Index comparison (real proof)

```bash
npm run db:index:compare
```

File: `04-index-comparison.ts`

This script:

1. Drops indexes
2. Runs `EXPLAIN (ANALYZE, BUFFERS)`
3. Creates indexes
4. Runs the same query again
5. Compares execution plans

It also:

* runs the same query using **TypeORM QueryBuilder**
* prints the SQL generated by TypeORM
* proves that ORM is not magic

---

### 5️⃣ Materialized view comparison

```bash
npm run db:view:compare
```

File: `05-materialized-view-comparison.ts`

Compares:

* heavy aggregation query
* materialized view read

Shows:

* lower execution time
* fewer buffer reads
* reduced CPU usage

The materialized view is accessed as a **read-only TypeORM entity**.

## 🧱 Migrations (Materialized View)

### Create migration

```bash
npx typeorm migration:create src/migrations/CreateProductSalesMaterializedView
```

### Run migrations

```bash
npx typeorm-ts-node-commonjs migration:run -d src/data-source.ts
```

Materialized views are:

* created via migrations
* refreshed explicitly
* never managed by `synchronize`

---

## 🚫 Best Practices

* ❌ `synchronize: true` in production

* ❌ creating indexes inside application logic

* ❌ trusting ORM without inspecting SQL

* ❌ measuring performance with `console.time`

* ✅ migrations for schema changes

* ✅ `EXPLAIN (ANALYZE, BUFFERS)` for performance

* ✅ explicit `REFRESH MATERIALIZED VIEW`

* ✅ indexes treated as infrastructure

---

## 🔍 Why `EXPLAIN (ANALYZE, BUFFERS)`?

Because application timings lie.

Only the database can tell you:

* how many rows were scanned
* which join strategy was used
* how many disk pages were read
* where the real bottleneck is
